# AI Platform — Architecture (SSOT)

Документ описывает **текущую реализацию** в репозитории и **целевую SaaS-архитектуру**. Исполняемый код здесь не приводится.

---

## CURRENT STATE

### STORAGE LAYER

- **PostgreSQL** (primary datastore).
- **Prisma** (ORM, миграции, типобезопасный доступ из `apps/api`).
- Файловое **JSON-хранилище** для сущностей SaaS **удалено и не используется**; данные пользователей, ключей, ассистентов, знаний, usage и rate-limit состояния — в таблицах БД.

### HTTP API (Fastify)

| Метод и путь | Auth | Назначение |
|--------------|------|------------|
| GET `/health` | нет | Статус процесса + проверка доступности Ollama |
| POST `/users` | нет | Создание пользователя: `id`, `email` |
| POST `/api-keys` | нет | Выдача ключа вида `sk-…`, привязка к `userId` |
| GET `/assistants` | Bearer | Список ассистентов текущего пользователя |
| POST `/assistants` | Bearer | Создание ассистента: `name`, `model`, `systemPrompt`, `userId` из ключа |
| POST `/knowledge` | Bearer | Добавление записи знаний: `assistantId`, `content` (строка) |
| GET `/usage` | Bearer | Агрегаты по использованию для пользователя |
| POST `/chat` | Bearer + rate limit | Диалог: `assistantId`, `message` → ответ Ollama |

### Пользователи и ключи

- Пользователь: **UUID** + **email** (поля `createdAt` в коде создания нет).
- API key: секрет передаётся клиентом один раз при создании; в БД хранится только **хэш** (`keyHash`). Отдельных полей `limit` / `usage` на ключе в текущей схеме **нет** (лимиты — отдельно, см. rate limit).

### Ассистенты

- Поля: `id`, `name`, `model`, `systemPrompt`, `userId`.
- Значение `model` может быть **`auto`** — тогда выбор модели делает отдельный слой (см. Model layer).

### Знания («RAG» в текущем виде)

- Запись: `id`, `assistantId`, `content` (одна строка текста).
- **Нет** типов источника (file/url), **нет** чанков, **нет** эмбеддингов, **нет** векторного поиска.
- В `POST /chat` к промпту подмешиваются до **трёх** записей по `assistantId`; содержимое **склеивается** в один блок текста и вставляется в структурированный промпт (SYSTEM / KNOWLEDGE / USER). Это **не** retrieval по смыслу, а ограниченная подстановка текста.

### Чат

- Один запрос — один вызов Ollama `generate`, ответ возвращается клиенту.
- **Нет** полноценной цепочки диалога в `POST /chat`: история реплик **не** подмешивается в промпт (только текущее `message` плюс system/knowledge). Таблицы `conversations` / `messages` в схеме могут быть заделом под целевой сценарий из раздела TARGET.

### Usage

- Append-события: `userId`, `model`, `tokens` (оценка по длине сообщения), `timestamp`.
- GET `/usage` отдаёт агрегаты по пользователю и разбивку по моделям.

### Rate limit

- Только на **`POST /chat`**: окно **1 минута**, **10** запросов на API key (константы в middleware).
- Состояние счётчиков — в PostgreSQL, таблица **`rate_limit_state`** (ключ по хэшу API key), через Prisma.

### Ollama

- URL из переменной окружения **`OLLAMA_URL`** (GPU-сервер в прод по `DEPLOYMENT_RULES.md`).
- Генерация: HTTP к Ollama API `generate`, не streaming.

### Widget

- Клиентский скрипт в `apps/widget/`, обращение к API с Bearer-ключом и идентификаторами ассистента (детали — в README виджета / деплое). Сервер виджета не отдельный микросервис — тот же backend за nginx.

### Model layer (текущий)

- **`modelRouter`**: при `model === "auto"` эвристика по тексту сообщения (ключевые слова, длина) выбирает имя модели Ollama.
- **`ensureModelAvailable`**: запрос списка моделей к Ollama (`/api/tags`), проверка наличия выбранной модели, **fallback** на фиксированный список имён либо первую доступную модель из ответа.

## PROBLEMS (остаточные к TARGET)

| Тема | Факт / последствие |
|------|-------------------|
| Persistence слоя | **PostgreSQL + Prisma** внедрены; JSON file storage **снят**. |
| Нет agents | В коде нет сущности «агент» отдельно от ассистента. |
| Нет истории диалогов в chat | `POST /chat` не создаёт conversation/message records для цепочки реплик. |
| Нет полноценного RAG | Нет chunking, embeddings, vector search — только подстановка до N строк знаний в промпт. |
| API keys / квоты | Нет квот на ключ в данных; rate limit на чат по ключу, не связан с «лимитом тарифа». |
| Multi-tenancy / security | Правила TARGET ниже обязательны; проверки ownership в коде должны покрывать каждый ресурс по ID. |
| Потоки (flow) | Нет зафиксированной цепочки шагов для chat / billing / событий — риск расхождения реализации между сервисами. |

## TARGET ARCHITECTURE

Целевая модель данных для SaaS: реляционная или документная **БД** с явными связями и миграциями (конкретный движок — решение инфраструктуры, не этот документ).

### MULTI-TENANCY (MANDATORY)

| Правило | Содержание |
|---------|------------|
| Идентификатор арендатора | Каждая бизнес-сущность либо имеет **`userId`**, либо связана через FK с сущностью, у которой **`userId`** (например Knowledge → Assistant → `userId`). |
| Фильтрация | Любой **SELECT/UPDATE/DELETE** по данным пользователя выполняется с **`WHERE userId = :resolvedUserId`** (или эквивалент через join), без «глобальных» выборок. |
| Доступ к чужим данным | **Запрещён.** Идентификаторы из тела запроса (`assistantId`, `conversationId`, …) **не** считаются доверенными: сопоставление с `userId` из API key обязательно. |
| Операция | Любая мутация → проверка **ownership**: ресурс принадлежит тому же `userId`, что и ключ. |

**Запрещено:**

- читать или изменять данные **без** привязки к разрешённому **`userId`** (включая админ-пути вне явной политики RBAC);
- **доверять** полям из клиента (`userId`, `email`, «чужой» `assistantId`) без проверки на сервере по ключу сессии / API key.

### API KEYS

| Поле | Тип смысла |

| usage | накопленное потребление или ссылка на агрегаты |

### ASSISTANTS

| Поле | Тип смысла |
|------|------------|
| id | идентификатор |
| userId | владелец |
| name | имя |
| model | имя модели или `auto` |
| systemPrompt | системный промпт |
| settings | JSON-настройки UI/поведения |

### AGENTS (AUTONOMOUS SYSTEM)

**AGENT** — автономная система выполнения (не путать с Assistant: отдельная таблица, отдельная бизнес-логика).

| Поле | Тип смысла |
|------|------------|
| id | идентификатор |
| userId | владелец (tenant) |
| name | имя |
| type | `chat` / `automation` (или расширяемый enum) |
| trigger | что запускает агента (расписание, webhook, событие, ручной старт) |
| tools | ссылки на записи **TOOLS** (`toolId[]`) или эквивалент связи один-ко-многим |
| flow | описание шагов сценария (steps): порядок, условия ветвления |
| memory | политика и хранилище памяти агента (краткосрочная / долгосрочная — отдельное проектирование) |

#### AGENT FLOW (логический порядок)

1. **trigger** — срабатывание условия старта  
2. **input** — нормализованный вход (пользователь, payload, контекст события)  
3. **tools execution** — вызовы разрешённых инструментов по политике  
4. **decision** — ветвление, остановка, повтор, эскалация  
5. **output** — результат пользователю, в очередь, во внешнюю систему  

**Правило:** ассистент и агент — **разные** сущности; не смешивать таблицы и бизнес-логику.

### TOOLS

Инструменты, которыми владеет агент (связь **Agent → Tools**).

| Поле | Тип смысла |
|------|------------|
| id | идентификатор |
| agentId | владелец-агент |
| type | `http` / `function` / `webhook` |
| config | URL, секреты по ссылке на vault, схема параметров, таймауты |

Связь: **один Agent — много Tools**; вызов только инструментов, привязанных к этому `agentId`, с проверкой **`userId`** агента.

### KNOWLEDGE (RAG)

| Поле | Тип смысла |
|------|------------|
| id | идентификатор |
| assistantId | привязка к ассистенту |
| type | `text` / `file` / `url` |
| content | текст или ссылка на объект хранилища |
| embeddingId | ссылка на вектор в vector store |

### CONVERSATIONS

| Поле | Тип смысла |
|------|------------|
| id | идентификатор |
| assistantId | контекст ассистента |
| userId | владелец / tenant |
| createdAt | время создания |

**Контракт `POST /chat` (целевой):** в теле запроса — опциональный **`conversationId`**.

| Ситуация | Действие сервера |
|----------|------------------|
| `conversationId` **отсутствует** или пустой | Создать новую запись **CONVERSATION** для пары (`userId`, `assistantId`), затем записать сообщения. |
| `conversationId` **передан** | Загрузить диалог; проверить **ownership** (`conversation.userId` = `userId` из ключа, `assistantId` совпадает); продолжить цепочку сообщений. |

### MESSAGES

| Поле | Тип смысла |
|------|------------|
| id | идентификатор |
| conversationId | поток |
| role | `user` / `assistant` (и при необходимости `system`) |
| content | текст |
| tokens | учёт для биллинга |

### USAGE

| Поле | Тип смысла |
|------|------------|
| id | идентификатор |
| userId | кто потребил |
| apiKeyId | какой ключ инициировал списание (для биллинга и лимитов) |
| model | какая модель |
| tokens | объём |
| cost | денежная метрика при наличии тарифов |

```

Assistant → Knowledge
```

## API DESIGN (целевое)

Минимальный контракт для SaaS (префикс версии `/v1` — по политике продукта):

| Операция | Назначение |
|----------|------------|
| POST `/assistants` | Создать ассистента |
| GET `/assistants` | Список ассистентов пользователя |
| POST `/agents` | Создать агента |
| POST `/chat` | Сообщение: `assistantId`, `message`, опционально **`conversationId`** (см. **CONVERSATIONS**) |
| GET `/conversations` | Список / история диалогов |
| POST `/knowledge` | Загрузка знаний (метаданные + последующая индексация) |

Аутентификация: Bearer API key → `userId`; авторизация: проверка владения ресурсом на каждом запросе.

## CHAT FLOW (целевой)

Линейный порядок обработки **одного** запроса к чату (ассистент):

1. **Validate user** — из API key получить `userId`; отклонить при невалидном ключе.  
2. **Load assistant** — по `assistantId` загрузить сущность; проверить **`assistant.userId` = `userId`**.  
3. **Load conversation** — по опциональному `conversationId`: создать новую беседу или загрузить существующую с проверкой ownership (см. **CONVERSATIONS**).  
4. **Retrieve knowledge (RAG)** — выборка релевантных чанков из vector store, не доверяя клиенту составу знаний.  
5. **Build prompt** — SYSTEM + retrieved knowledge + история сообщений (если включена) + текущее user message.  
6. **Call model** — вызов провайдера через model layer (routing, fallback).  
7. **Save message** — запись user и assistant реплик в **MESSAGES** (и обновление метаданных **CONVERSATIONS** при необходимости).  
8. **Update usage** — атомарно: токены, при необходимости `apiKeyId`, для биллинга и лимитов.

## BILLING RULES (целевые)

| Правило | Содержание |
|---------|------------|
| Учёт по ключу | **Usage** и rate-limit-бюджеты привязываются к **API key** (`apiKeyId`), не только к `userId` в абстракции. |
| Лимиты на ключ | Квоты (запросы/минута, токены/день) хранятся на уровне ключа или тарифного плана, ссылающегося на ключ. |
| Токены → деньги | Таблица тарифов: модель × цена за 1K токенов → поле **`cost`** в **USAGE** или отдельный биллинг-пайплайн. |

**Связь:** **API KEY** → множество записей **USAGE** (линия списания по каждому значимому вызову).

Исключение утечки: агрегаты биллинга доступны только владельцу ключа / аккаунта после проверки **`userId`**.

## SECURITY (целевой)

| Требование | Содержание |
|------------|------------|
| Auth на endpoint’ах | Все операции с данными пользователя — с **аутентификацией** (исключения: явно перечисленные публичные маршруты вроде health по политике продукта). |
| Ownership | Каждый доступ к объекту по ID — проверка принадлежности **`userId`**. |
| Rate limit | Лимиты на ключ и/или на пользователя; единая политика HTTP **429**. |
| API key | Хранить **хэш** ключа или секрет в vault; в БД — не plaintext `sk-…`. |
| Логирование | Аудит: ключ (префикс), `userId`, идентификаторы сущностей, без утечки тел промптов в незащищённые логи по умолчанию. |

## RAG (целевое, правильное)

| Этап | Содержание |
|------|------------|
| Ingest | Источники: текст, файл, URL; нормализация в текст. |
| Chunking | Разбиение на фрагменты с перекрытием и метаданными (источник, позиция). |
| Embeddings | Векторизация чанков провайдером эмбеддингов (локально или API). |
| Vector DB | Хранение в **Qdrant** или **pgvector** (или эквивалент): коллекция на ассистента / tenant. |
| Retrieval | Запрос пользователя → top-k похожих чанков → контекст в промпт с лимитом токенов. |

Plain text в БД без эмбеддингов для поиска по смыслу **не** считается завершённым RAG.

## MODEL LAYER (целевое)

| Тема | Содержание |
|------|------------|
| Auto routing | Правила или малая модель-маршрутизатор: выбор модели по задаче/стоимости. |
| Fallback | При недоступности модели — очередь альтернатив и деградация с метрикой. |
| Providers | Ollama как один провайдер; опционально облачные API; единый интерфейс `complete` / `embed`. |
| Cost control | Верхние границы стоимости на запрос / на пользователя; отказ или cheaper model при превышении бюджета. |
| Latency control | Таймауты, отмена долгих вызовов, выбор быстрой модели для интерактивных сценариев. |
| Priority routing | Очереди: платный тариф / SLA → приоритет в обработке при нагрузке (политика продукта). |
| Fallback chain | Упорядоченный список моделей: primary → secondary → … до успеха или явной ошибки; метрики на каждом шаге. |

Текущий `modelRouter` + `ensureModelAvailable` в коде — задел под routing/fallback; перенос в конфигурацию, наблюдаемость и cost/latency — эволюция под эту таблицу.

## EVENTS (целевые)

Асинхронные или внутренние события для аналитики, биллинга и вторичных процессов:

| Событие | Назначение |
|---------|------------|
| `message.created` | Запись сообщения, обновление диалога, триггеры RAG/памяти. |
| `assistant.used` | Учёт использования ассистента (метрики продукта). |

Публикация: шина событий или outbox в БД — решение инфраструктуры; потребители не нарушают **MULTI-TENANCY** (фильтр по `userId` в обработчиках).

## ЗАПРЕТЫ (целевой прод)

- **Нельзя** возвращаться к **файловому хранилищу** (в т.ч. JSON-файлам в репозитории) как к долгосрочному хранилищу SaaS.
- **Нельзя** хранить «знания» только как **единый plain text** без чанков и векторного индекса, если заявлен полноценный RAG.
- **Нельзя** смешивать сущности **assistants** и **agents** в одной таблице без явного разделения типа и поведения.
- **Нельзя** обходить проверку **`userId`** / ownership: любой ID из запроса проверяется на сервере.
